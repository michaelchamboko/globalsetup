# PRD Review Checklist

## Completeness

- [ ] Problem statement is specific and describes the current pain point
- [ ] Target users are identified with concrete descriptions
- [ ] Every requirement has a unique ID
- [ ] Every requirement is specific and measurable (no vague language)
- [ ] Priority is assigned to each requirement (must have / should have / nice to have)
- [ ] Constraints are documented
- [ ] Non-goals are explicitly stated
- [ ] Success metrics have target values and measurement methods
- [ ] Dependencies are identified
- [ ] Stakeholder approval is recorded

## Language Quality

- [ ] No ambiguous words: "should," "might," "could consider," "possibly," "maybe"
- [ ] No undefined terms or acronyms
- [ ] No contradictory requirements
- [ ] No requirements that overlap or duplicate each other

## Feasibility

- [ ] Requirements are technically feasible within constraints
- [ ] Timeline is realistic for the scope
- [ ] Dependencies are available or have confirmed delivery dates
- [ ] No requirement depends on an undefined external system

## Testability

- [ ] Every requirement can be verified with a test or acceptance criterion
- [ ] Success metrics can be measured with existing or planned instrumentation
